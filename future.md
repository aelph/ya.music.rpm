# Развитие проекта

Скрипт проверен и многократно опробован.

Дальнейшим развитием могло бы быть изменение механизма обновления, как встроенной функции программы.

## Итерация 1

---

### Step 1

> Изучи проект. Внутри пересобранного RPM-пакета, есть механизм обновления программы, который работает только в DEB-версии.
> У нас есть скрипт, который способен обновить программу при помощи пересборки RPM из DEB.
> Возможно ли сделать так, чтобы встроить наш скрипт в механизм обновления программы?
> Оцени работу и ответь. Ничего пока не меняй.

Встроить скрипт обновления в механизм обновления возможно, но только через 
- патч внутреннего `app.asar`;
- или через собственный update-feed для RPM. 
Без этого никак.

Почему так:
- В дереве сборки тип пакета выставлен как `deb`: [package-type](/ya.music.rpm/build/yandexmusic-5.95.0/opt/Яндекс Музыка/resources/package-type:1).
- Апдейтер смотрит на этот тип и выбирает `DebUpdater` (подтверждено разбором `app.asar`).
- `DebUpdater` выполняет `dpkg -i ... || apt-get install -f -y` (подтверждено разбором `app.asar`), поэтому в RPM-среде сломается.
- Источник обновлений задан как generic URL: [app-update.yml](/ya.music.rpm/build/yandexmusic-5.95.0/opt/Яндекс Музыка/resources/app-update.yml:1).

Что реально сделать:
1. Патчить `DebUpdater` в `app.asar`, чтобы вместо `dpkg/apt` вызывался скрипт (передавать ему путь к скачанному `.deb`).
2. Либо перевести приложение на `RpmUpdater` (`package-type=rpm`) и поднять/подменить feed, где есть `.rpm`-артефакты и корректный `latest-linux.yml`.

Практический вывод:
- Технически: **да, возможно**.
- Архитектурно “чисто” без модификации приложения: **нет**.
- Наименьшие трудозатраты: патч `app.asar` + вызов вашего скрипта из `DebUpdater`.
- Риски: хрупкость при каждом новом релизе (патч надо повторять), возможные вопросы целостности/политик вендора.

### Step 2

> ну, а при вызове скрипта в патче `app.asar` скрипт может рекурсивно сам себя пропатчить в новом rpm при сборке?

Да, может. Но это должно быть не «магическое самопатчение», а повторяемый шаг в пайплайне сборки RPM.

Коротко как это работает:

1. Патч в `app.asar` вызывает ваш updater-скрипт.
2. Скрипт берёт новый `.deb`, пересобирает RPM.
3. Во время этой пересборки **снова** применяется патч `app.asar`.
4. Установленный новый RPM уже содержит тот же хук, и цепочка сохраняется на следующих обновлениях.

Что важно, чтобы не словить рекурсию/поломку:

- Патч должен быть идемпотентным (проверка маркера «уже пропатчено»).
- Нужен guard от повторного запуска (`UPDATE_IN_PROGRESS=1`/lockfile).
- Скрипт должен жить в стабильном пути, который есть после обновления (лучше класть в пакет, а не ссылаться на каталог репо).
- В `build_rpm.sh` сейчас патчится только `.desktop`; для этой схемы надо добавить явный шаг патча `app.asar` при каждой сборке.

Итог: **да, возможно и это рабочая стратегия**.

### Step 3

> то есть оба скрипта нужно засунуть в пакет.
модифицировать тоже оба: build в части явного шага `app.asar`, а install в части использования `/tmp/` для файлов сборки. так?

Да, именно так.

Нужна такая схема:

1. Оба скрипта включить в RPM (в стабильный путь, например `/opt/Яндекс Музыка/updater/` или `/usr/libexec/yandexmusic-updater/`).
2. `build`-скрипт доработать: явный и идемпотентный шаг патча `app.asar` при каждой пересборке.
3. `install`-скрипт доработать: рабочие артефакты сборки складывать в `/tmp` (лучше через `mktemp -d`) и чистить за собой.
4. Вызов из патча `app.asar` должен идти на путь скрипта из установленного пакета, а не из каталога репозитория.
5. Добавить lock/guard от повторного запуска, чтобы не поймать зацикливание.

Это правильное направление.

### Проблемы, выявленные в попытке реализации первой итерации (апрель 2026)

1. Патч `app.asar` оказался критически чувствительным к формату вставки.
   На одном из этапов в `DebUpdater.js` попали буквальные `\n`, что дало `SyntaxError` в main process и приложение перестало запускаться.

2. Недостаточно просто менять файл внутри `app.asar`.
   После модификации нужно пересчитывать `integrity` в ASAR header (SHA256 + blocks), иначе возникает несоответствие целостности.

3. Даже при корректном пересчёте ASAR integrity UI всё равно может ломаться.
   В бинарнике обнаружены fuse-флаги Electron (`IsEmbeddedAsarIntegrityValidationEnabled`, `IsOnlyLoadAppFromAsarEnabled`), поэтому изменение `app.asar` в принципе может быть неподдерживаемым для этой сборки.

4. Ветка установки через `pkexec` должна быть «мягкой» при отказе пользователя.
   Без `try/catch` в `DebUpdater` отказ авторизации приводил к ошибке апдейтера и нестабильному поведению после закрытия.

5. `install_rpm.sh` в root-контексте не должен повторно требовать `sudo`.
   Запуск через `pkexec` + внутренний `sudo dnf ...` давал риски срыва в неинтерактивном окружении.

6. Блокировка через lock-directory оказалась хрупкой.
   При аварийных прерываниях оставался «мертвый» lock, из-за чего повторный запуск обновления блокировался.
   Более надёжно использовать `flock` на lock-file.

7. Сборка RPM имела технические сбои пайплайна.
   `rpmbuild` падал на `Installed (but unpackaged) ...spec`, когда `.spec` оставался внутри `buildroot`; это нужно обходить вынесением `.spec` во временный файл вне buildroot.

Тестовая ветка с этой реализацией и промежуточными правками:
- `wip_updater_intermediate_2026_04_19` (коммит `db4ff16`)

## Куда двигаться дальше

1. Признать, что путь с патчем `app.asar` сейчас тупиковый.
- Даже при корректном `integrity` UI ломается.
- В бинарнике есть fuse-ограничения (`EmbeddedAsarIntegrityValidation`, `OnlyLoadAppFromAsar`).

2. Зафиксировать текущую попытку как исследовательскую.
- Вся история уже в ветке `wip_updater_intermediate_2026_04_19` (`db4ff16`).

3. Перейти на «чистую» архитектуру без модификации `app.asar`.
- Либо свой RPM update-feed и переключение на `RpmUpdater`/`package-type=rpm`.
- Либо полностью внешний updater (отдельный сервис/скрипт), не встроенный в внутренний DebUpdater.

4. Если оставаться на внешнем updater:
- Скрипты в пакет (`/opt/.../updater/`),
- `mktemp` в `/tmp`,
- `flock`-блокировка,
- отсутствие `sudo` внутри root-контекста,
- мягкая обработка отказа в `pkexec`.

Итого: следующий шаг не «чинить патч в asar», а проектировать обновление вне `asar`-модификаций.
